## CRITICAL: Data Accuracy Problem (MUST FIX)

### The Problem
The mockup shows inaccurate completion data. Habits like "Complete 3 MIT Tasks", "Deep Work 60m/15m", "Journal: Reflect & Plan" show as 100% completed for 87 consecutive days. Lucas says this is WRONG. The data extraction from the Streaks iOS app backup is miscounting completions.

### Source Data
- **File**: `~/Library/Mobile Documents/iCloud~com~streaksapp~streak/Documents/Mac - 20260223T051921 - scheduled.streaks`
- **Format**: JSON with a `tasks` array. Each task has a `log` array of entries.
- **Date range**: ~Nov 25, 2025 to Feb 22, 2026 (90 days)

### Streaks Backup Format (What We Know)

#### Task fields
- `st`: Status. `N` = active (show these), `A` = archived (hide these)
- `n`: Name of the habit
- `cat`: Category
- `log`: Array of log entries

#### Log entry `t` field (entry type)
- `t=1`: Manual completion (user tapped "done"). **This is a real completion.**
- `t=2`: Manual failure (user tapped "missed"). **This is a real miss.**
- `t=4`: Auto-backfill. **THIS IS THE PROBLEM.** These entries are automatically generated by Streaks to fill in gaps. They do NOT represent actual user action. They should likely be treated as "no data" (-1), NOT as completions.
- `t=5`: Retroactive entry. Check if it has a `ts` (timestamp) field. If `ts` exists, it's a real entry. If no `ts`, treat as no data.
- `t=15`: HealthKit data sample. Has a `p` field with the actual numeric value (protein grams, water oz, calories, etc.)

#### The Bug
The previous data extraction script treated t=4 (auto-backfill) entries as completions. This is why habits show artificially high completion rates. Streaks auto-backfills days, but that doesn't mean Lucas actually did the habit.

### How to Fix
1. Re-extract data from the Streaks backup
2. Only count t=1 and t=5 (with ts) as completions
3. Count t=2 as misses
4. Count t=4 and t=5 (without ts) as no data (-1)
5. For t=15 (HealthKit): use the `p` field value. Completion is derived by comparing `p` against the task's target.
6. Days with NO log entry at all = no data (-1)

### What "Correct" Data Looks Like
- Habits like "Brush Teeth" probably do NOT have 87/87 days completed. Some of those are t=4 auto-backfills.
- The overall 88% completion rate is almost certainly inflated.
- The real completion rate is probably significantly lower, which is actually what Lucas WANTS to show (accountability means showing the truth).

## Design Decisions Made (via Design Boardroom)

### Color Palette (approved)
- Green (done): `#3FB950`
- Yellow (close/near target): `#D2A04A`
- Red (missed): `#DA3633` at full opacity (no transparency)
- No data: `#13161b` (near-invisible)
- Background: `#0d1117`
- Surface: `#161b22`

### Layout (approved)
- **Dates reversed**: most recent on the LEFT (visible immediately on mobile without scrolling)
- **Week separator lines**: vertical borders between ISO weeks (Monday start)
- **Week labels**: rotated vertical text showing week start date ("Feb 16", "Feb 9", etc.)
- **Habit names**: 180px desktop / 140px mobile max-width
- **Category headers**: slim (9px uppercase, minimal padding)
- **Habit sort order**: within each category, habits with mixed results (some misses) appear first; solid green walls anchor the bottom

### Filters (performance-based, not category-based)
1. **All** - default view, all habits, all 90 days
2. **This Week** - last 7 days only
3. **On Fire** - habits with active streaks, sorted by streak length
4. **Trending Up** - habits where recent completion rate beats earlier period
5. **Needs Work** - habits below 50% completion rate
- Each filter shows a one-line description when selected

### Inline Numeric Values
- HealthKit habits (protein, water, calories) show compact values inside wider cells (30px)
- Format: protein -> "215" (grams), water -> "49" (oz), calories -> "2.7k" (kcal)
- Full details in tooltip on tap/hover
